# Паттерны использования ХаниМани

Синтез из официальных источников hmbee.ru/ru/docs/\* — конкретные пользовательские сценарии и UX-решения, которые информируют проектирование hmoney.

## Концептуальный фреймворк: 3 горизонта

ХаниМани строится вокруг трёх временных горизонтов:

| Горизонт | Вопрос | Инструмент |
|----------|--------|-----------|
| **Вчера** — учёт | Куда ушли деньги? | Список транзакций, Отчёты |
| **Сегодня** — контроль | Сколько тратить / не трогать / отложить? | Конверты, Прогноз Резервов |
| **Завтра** — планирование | Хватит ли денег? Когда достигну цель? | Плановые транзакции, Цели |

Прогноз Резервов — ключевой механизм горизонта «Сегодня/Завтра». Горизонт расчёта: **до 12 месяцев вперёд**.

## Онбординг: с чего начать

По рекомендации HM (docs/quick_start):

1. Создать все активные счета с **текущим балансом**
2. Запланировать регулярные доходы и расходы (зарплата, аренда, продукты)
3. Проверить прогноз Резервов — если дефицит, перевести деньги из ХаниМани
4. Начать вносить транзакции — ежедневно или раз в несколько дней
5. Самостоятельно создать фонд **«На чёрный день»** (первый пользовательский конверт)

Базовая система: ХаниМани + Резервы (автосозданы) + «На чёрный день» (создаётся вручную).

> EN-14: после регистрации предлагать создать фонд «На чёрный день» как онбординг-подсказку.

## Переводы между конвертами

**Пример: фрилансер и налоги** (docs/saving_for_taxes):

1. Создать фонд «Налоги»
2. Запланировать **ежемесячный перевод** из Резервов в фонд (13% от среднего дохода)
3. Когда пришло время платить — расход с фонда «Налоги»
4. Излишек или недостаток — корректировка через переводы

Переводы конверт→конверт учитываются в прогнозе Резервов.

> EN-13: перевод между конвертами — отдельная операция, аналог TR-02 transfer для конвертов.

## Учёт кредитных карт

- Баланс кредитки **начинается с 0** (ничего не должны)
- Каждая покупка делает баланс отрицательным (долг банку)
- **Оплата кредитки = перевод** между счетами (дебетовый → кредитный), **не расход**
- Расход уже был зафиксирован при покупке

## Учёт долгов

Два подхода в зависимости от размера:

| Ситуация | Решение |
|----------|---------|
| Крупный займ (ипотека) | Регулярный расход с датой окончания |
| Мелкий разовый долг | Расход (или доход), парный запланированный доход (или расход) |
| Частые долги с одним человеком | Отдельный счёт на этого человека |
| Общий учёт долгов | Два счёта: «Дал(а) в долг» и «Взял(а) в долг» |

Флаг «Исключить из баланса» на долговом счёте убирает его из суммы конверта ХаниМани.

## Возврат после перерыва

Два варианта (docs/comeback):

**Быстрый старт заново** — сброс всех данных, пересчёт балансов счетов.

**Сохранить историю**:
1. Отредактировать балансы всех счетов до текущих значений
2. Массово удалить все старые неподтверждённые плановые транзакции
3. При желании — добавить по одному суммарному расходу на каждый пропущенный месяц для статистики

> TR-13: bulk delete неподтверждённых транзакций — важная функция для «возврата».

## Ускорение ввода (UX-паттерны)

Из docs/faster — три принципа, совместимые с нашим дизайном:

| Принцип | Детали | Соответствие в hmoney |
|---------|--------|----------------------|
| **Округление** | 349.60 → 350 руб. | Целые рубли — уже реализовано |
| **Объединение** | Два магазина, одна категория → одна транзакция | Natural language: `350 продукты, 200 кофе` |
| **Плановые дни** | Все еженедельные расходы — по воскресеньям | Плановые (mode: plan) + дата |
