# Реализация аутентификации

## Выбор сервиса

**Firebase Auth** (email/password) — бесплатно, безлимитно, минимальная конфигурация.

Альтернатива: Supabase Auth — функционально богаче (magic link, OAuth), но требует больше конфигурации и серверной инфраструктуры.

## Архитектура

```
BrowserRouter
├── /login      → LoginPage        (public)
├── /register   → RegisterPage     (public)
├── /           → AuthGuard → Dashboard  (protected)
└── *           → редирект на /
```

### Компоненты

| Компонент | Назначение |
|-----------|-----------|
| `AuthGuard` | Проверяет `user` из `authStore`. Если `loading` — показывает спиннер. Если нет `user` — редирект на `/login`. |
| `LoginPage` | Форма: email + пароль. Ошибки: неверный email/пароль, пользователь не найден. |
| `RegisterPage` | Форма: email + пароль + подтверждение. Валидация: совпадение паролей, длина ≥ 6. После успеха — создаётся документ `users/{uid}` в Firestore. |

### Состояние (Zustand)

`authStore`:
- `user: User | null` — текущий пользователь Firebase
- `loading: boolean` — true пока `onAuthStateChanged` не ответил
- `login(email, password)` — `signInWithEmailAndPassword`
- `register(email, password)` — `createUserWithEmailAndPassword` + создание `users/{uid}`
- `logout()` — `signOut`

`onAuthStateChanged` инициализируется при создании store (один раз).

### Firebase config

Ключи читаются из `import.meta.env.VITE_FIREBASE_*` (файл `.env` в gitignore).

```env
FIREBASE_API_KEY=...
FIREBASE_AUTH_DOMAIN=...
FIREBASE_PROJECT_ID=...
FIREBASE_STORAGE_BUCKET=...
FIREBASE_MESSAGING_SENDER_ID=...
FIREBASE_APP_ID=...
```

## SPA routing на GitHub Pages

При деплое на GitHub Pages все пути (кроме `/`) должны вести на index.html.
Решено копированием `dist/index.html → dist/404.html` после сборки.

## Безопасность

- Firestore security rules будут добавлены на этапе работы с данными
- Email/password передаются по HTTPS (Firebase)
- `.env` в `.gitignore` — ключи не попадают в репозиторий
