# Техническое задание: hmoney

## 1. Назначение

Разработать веб-приложение (SPA + PWA) для ручного учёта личных финансов — упрощённый аналог HoneyMoney (https://hmbee.ru).

Приложение предназначено для одного пользователя (личное использование), с возможностью расширения на семью.
Домен: https://money.vitalik.dev

## 2. Целевая аудитория

- Люди, которые хотят вести учёт личных финансов вручную
- Люди, ценящие осознанное отношение к деньгам
- Пользователи, не желающие подключать банковские счета к приложениям

## 3. Основные функции (MVP)

### 3.1 Управление транзакциями
- Создание, редактирование, удаление транзакций
- Поля: дата, сумма, категория, описание, счёт, конверт
- Типы: доход, расход, перевод; неподтверждённые — статус (фильтр в списке)
- Режимы: Факт (реальная) и План (запланированная)
- Natural language input: `5*250 яблоки (комментарий)` — парсинг количества × цена
- Поддержка регулярных (recurring) транзакций
- Фильтрация по счетам, конвертам, типам, категории, описанию

### 3.2 Календарь
- Просмотр транзакций в календарном формате (недели × дни)
- Быстрое добавление транзакции кликом на день
- Сводка доходов/расходов/разницы за месяц + % не потрачено
- Net Worth в хедере на всех страницах
- Статистика расходов/доходов (ALT+Z)
- Drag-and-drop копирование транзакций (зажат Alt/Ctrl)

### 3.3 Счета
- Типы: card (creditLimit), investment (group=investments), cash (по умолчанию)
- Отслеживание баланса по каждому счёту
- Кредитный лимит для кредитных карт
- Флаг исключения из общего баланса
- Группировка: Favorites, Investments, Hidden

### 3.4 Конверты (Envelopes)

#### Системные конверты (всегда присутствуют, ровно по одному)
- **ХаниМани** (`spending`) — свободные деньги на текущие траты
- **Резервы** (`reserve`) — зарезервировано на плановые расходы

Создаются автоматически при регистрации. Нельзя создать повторно или удалить.

#### Пользовательские конверты (0..20 суммарно)
Два подтипа:
- **Фонды** (`fund`) — накопительные, целевая сумма опциональна
- **Цели** (`goal`) — накопительные, целевая сумма обязательна

Пользователь создаёт, редактирует и удаляет. Лимит: **20 конвертов суммарно** (Фонды + Цели).

**Прогноз Резервов**: система автоматически рассчитывает дефицит/профицит на основе:
- Текущего баланса Резервов
- Плановых доходов и расходов на период
- Запланированных переводов между конвертами
На горизонте до **12 месяцев вперёд**. При дефиците — пользователь переводит деньги из ХаниМани в Резервы.

**Переводы между конвертами**: пользователь может переводить сумму из одного конверта в другой (например, из Резервов в фонд «Налоги»). Используется для накоплений с заданным ритмом.

### 3.5 Базовая аналитика
- Доходы/расходы/разница за период
- Сравнение периодов
- Net Worth отчёт
- Прогноз Резервов (дефицит/профицит)

### 3.6 Горячие клавиши
- Ctrl+Enter — сохранить транзакцию
- Esc — закрыть модал
- Alt+1..4 — навигация по разделам
- Alt+←/→ — переключение месяца
- Alt+↑ — текущий месяц
- Alt+Z — статистика
- Alt+X — прогноз Резервов (дефицит/профицит)

## 4. Технические требования

### 4.1 Платформа
- **Хостинг**: GitHub Pages
- **Домен**: https://money.vitalik.dev (CNAME → vitalikxp.github.io)
- **CI/CD**: GitHub Actions (авто-сборка и деплой при пуше в `master`)
- **Формат**: SPA (Single Page Application)
- **PWA**: поддержка офлайн-режима

### 4.2 Фронтенд
- **Язык**: TypeScript
- **Фреймворк**: React 19
- **Бандлер**: Vite
- **Стили**: Tailwind CSS v4
- **Роутинг**: React Router 7
- **AI-assisted development**: opencode.ai

### 4.3 Бекенд и данные
- **База данных**: Firebase Firestore (free tier)
- **Аутентификация**: Firebase Auth
- **Офлайн**: кэширование данных локально (PWA + Firestore offline persistence)

### 4.4 Ограничения free tier Firebase
- 1 GB хранимых данных
- 10 GB/мес загрузок
- 50k reads/day, 20k writes/day
- 10 MB/request

## 5. Ограничения по реализации

- Только ручной ввод (нет автоматической синхронизации с банками)
- Однопользовательский режим (MVP)
- Локализация — русский язык (MVP)
- Валюта — RUB (MVP)

## 6. План работ

1. **Этап 1: Инициализация проекта** — Vite + React + Tailwind + Firebase
2. **Этап 2: Аутентификация** — регистрация, вход, защита маршрутов
3. **Этап 3: Счета и Конверты** — CRUD счетов и 4 типа конвертов
4. **Этап 4: Транзакции** — CRUD транзакций, natural language input, Факт/План
5. **Этап 5: Календарь** — календарный просмотр, статистика
6. **Этап 6: Аналитика** — отчёты по месяцам, net worth
7. **Этап 7: PWA** — офлайн-режим, иконки, манифест
8. **Этап 8: Полировка** — горячие клавиши, UI/UX, тесты, деплой

## Вопросы и противоречия

> Вопросы ниже закрыты с принятыми решениями. Оставлены для истории.

- ✅ **План vs Факт**: `mode` определяет поведение. План-транзакции не влияют на баланс счетов — только на прогноз Резервов. Флаг `affectsBalance` не нужен.
- ✅ **Регулярные транзакции**: в MVP — ручное создание через шаблон. Cloud Functions — post-MVP.
